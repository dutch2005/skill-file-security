# 🛡️ Supabase Security Checklist

> Run before going live with a Supabase project.
> Generated by security-skill v1.0.0

---

## 🔴 Critical — Must Pass Before Launch

### Row Level Security (RLS)
- [ ] **RLS enabled on ALL tables**
  ```sql
  SELECT tablename, rowsecurity FROM pg_tables WHERE schemaname = 'public';
  -- Every table must show rowsecurity = TRUE
  ```
- [ ] Each table has at least one RLS policy
- [ ] Users can only access their own data
- [ ] Admin policies restricted (using `service_role` or custom claims)
- [ ] No policy with `USING (true)` that grants public access

### Service Role Key
- [ ] **`SUPABASE_SERVICE_ROLE_KEY` is NOT in frontend code**
- [ ] Service role key only used server-side (API routes, Cloud Functions)
- [ ] Service role key not in `.env.local` committed to git
- [ ] Anon key used for frontend (it's safe — RLS protects the data)

### Authentication
- [ ] Email confirmations enabled (if applicable)
- [ ] Redirect URLs allowlisted (not `*`)
  - Supabase Dashboard → Auth → URL Configuration
- [ ] Rate limiting enabled on auth endpoints
- [ ] Leaked password protection enabled (HaveIBeenPwned check)

---

## 🟠 High — Fix Before Launch

### Database Security
- [ ] PostgreSQL password is strong (auto-generated by Supabase is fine)
- [ ] Direct database connections from app use pooler (not direct connection)
- [ ] SSL enforced on database connections
- [ ] Database exposed ONLY to Supabase (not to public internet)

### API
- [ ] CORS configured to your domains only (not `*`)
  - Supabase Dashboard → Settings → API → CORS
- [ ] JWT secret is strong (auto-generated by Supabase is fine)
- [ ] API keys rotated if compromised

### Storage
- [ ] Buckets are private by default
- [ ] Public buckets only contain truly public content
- [ ] Storage policies restrict access to bucket per user
- [ ] File size limits configured in bucket settings

---

## 🟡 Medium — Fix Soon

### RLS Policy Quality
- [ ] Policies use `auth.uid()` not `auth.role()`
- [ ] Admin policies use a dedicated `admins` table or custom JWT claim
- [ ] Policies tested with multiple user scenarios

### Edge Functions (if used)
- [ ] Functions validate all inputs
- [ ] Functions use service role key securely
- [ ] Function endpoints not publicly listed

### Backups
- [ ] Point-in-time recovery enabled (Pro plan)
- [ ] Export strategy defined for free tier

---

## 🔵 Low / Info

- [ ] Project name doesn't expose sensitive info
- [ ] Custom domain configured
- [ ] `memory-security.md` tracks anon key rotation

---

## ✅ Quick SQL Checks

```sql
-- 1. All tables have RLS enabled?
SELECT tablename, rowsecurity
FROM pg_tables
WHERE schemaname = 'public' AND rowsecurity = FALSE;
-- Should return 0 rows

-- 2. All tables have at least one policy?
SELECT tablename FROM pg_tables WHERE schemaname = 'public'
EXCEPT
SELECT tablename FROM pg_policies WHERE schemaname = 'public';
-- Should return 0 rows

-- 3. Any permissive "allow all" policies?
SELECT tablename, policyname, qual
FROM pg_policies
WHERE schemaname = 'public' AND qual = 'true';
-- Should return 0 rows (or review each one)
```
